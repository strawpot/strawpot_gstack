# Safety Guardrails

You have safety guardrails active. Before executing any command, mentally check it against the dangerous patterns below. If a command matches a dangerous pattern and does not match a safe exception, **stop and warn the user** before proceeding. Explain the risk, suggest the safe alternative, and ask for explicit confirmation.

These guardrails are not optional suggestions. A missed warning can cost hours of recovery or permanent data loss, while a false alarm costs five seconds of reading. Treat them as hard constraints unless the user explicitly overrides after seeing the warning.

## Dangerous patterns by category

### Filesystem

| Pattern | Example | Risk |
|---------|---------|------|
| `rm -rf` / `rm -r` / `rm --recursive` | `rm -rf /var/data` | Recursive delete, potentially unrecoverable |
| `chmod -R 777` | `chmod -R 777 /etc` | Opens all permissions recursively |
| `chown -R` on system dirs | `chown -R user /usr` | Breaks system ownership |
| `mv` or `cp` overwriting without backup | `mv config.yml config.yml.bak && mv new.yml config.yml` | Silent data overwrite |
| `> file` (redirect truncate) | `> /etc/hosts` | Empties file contents instantly |

**Safe alternatives:**
- Instead of `rm -rf`, move to a trash directory first: `mv target /tmp/trash-$(date +%s)/`
- Instead of blind `rm -r`, list contents first with `ls -la` or use `rm -ri` for interactive confirmation
- Before overwriting files, create a timestamped backup: `cp file file.bak.$(date +%s)`

**Safe exceptions (no warning needed):**
These are build artifacts and caches that are safe to delete recursively:
- `rm -rf node_modules`
- `rm -rf .next`
- `rm -rf dist`
- `rm -rf __pycache__`
- `rm -rf .cache`
- `rm -rf build`
- `rm -rf .turbo`
- `rm -rf coverage`
- `rm -rf tmp` or `rm -rf .tmp`
- `rm -rf target` (Rust/Java build output)

The key distinction: build artifacts are regenerated by the next build. User data, config, and system files are not.

This list covers common cases. If a project's `.gitignore` or build config indicates other regenerable output directories, treat those as safe exceptions too.

### Database

| Pattern | Example | Risk |
|---------|---------|------|
| `DROP TABLE` / `DROP DATABASE` | `DROP TABLE users;` | Permanent data loss |
| `TRUNCATE` | `TRUNCATE orders;` | Deletes all rows, no rollback in most engines |
| `DELETE FROM` without `WHERE` | `DELETE FROM customers;` | Wipes entire table |
| `ALTER TABLE DROP COLUMN` | `ALTER TABLE users DROP COLUMN email;` | Column data lost permanently |
| `UPDATE` without `WHERE` | `UPDATE users SET role = 'admin';` | Modifies every row |

**Safe alternatives:**
- Before `DROP TABLE`, verify with `SELECT COUNT(*) FROM table` and confirm the user understands the row count
- Instead of `TRUNCATE`, use `DELETE FROM table WHERE condition` with an explicit condition
- Before any destructive write, run the equivalent `SELECT` first to preview affected rows: `SELECT * FROM users WHERE condition` before `DELETE FROM users WHERE condition`
- Use transactions: `BEGIN; DELETE FROM ...; SELECT COUNT(*) ...; -- verify, then COMMIT or ROLLBACK`
- For schema changes, create a migration file instead of running raw DDL

### Git

| Pattern | Example | Risk |
|---------|---------|------|
| `git push --force` / `git push -f` | `git push -f origin main` | Rewrites remote history, other collaborators lose commits |
| `git reset --hard` | `git reset --hard HEAD~3` | Discards uncommitted work and rewrites local history |
| `git checkout .` / `git restore .` | `git checkout .` | Discards all unstaged changes |
| `git clean -fd` | `git clean -fd` | Deletes untracked files permanently |
| `git branch -D` | `git branch -D feature` | Force-deletes branch even if unmerged |
| `git rebase` on shared branches | `git rebase main` on a pushed branch | Rewrites shared history |

**Safe alternatives:**
- Instead of `git push --force`, use `git push --force-with-lease` (refuses if someone else pushed)
- Instead of `git reset --hard`, use `git stash` to save work first, or `git reset --soft` to unstage without losing changes
- Instead of `git checkout .`, use `git stash` to preserve changes reversibly
- Instead of `git clean -fd`, use `git clean -fdn` (dry run) first to preview what would be deleted
- Instead of `git branch -D`, use `git branch -d` (lowercase) which refuses to delete unmerged branches
- Before any history rewrite, create a backup ref: `git branch backup-$(date +%s)`

### Container orchestration

| Pattern | Example | Risk |
|---------|---------|------|
| `kubectl delete` | `kubectl delete pod my-pod` | Removes production resources |
| `kubectl delete namespace` | `kubectl delete namespace production` | Destroys entire namespace and all resources in it |
| `docker rm -f` | `docker rm -f my-container` | Force-kills and removes running container |
| `docker system prune` | `docker system prune -a` | Removes all unused images, containers, volumes |
| `docker volume rm` | `docker volume rm pgdata` | Deletes persistent data |
| `helm uninstall` | `helm uninstall my-release` | Removes deployed release |

**Safe alternatives:**
- Before `kubectl delete`, verify context with `kubectl config current-context` and confirm you are not targeting production
- Use `kubectl delete --dry-run=client` to preview what would be deleted
- Instead of `docker system prune -a`, use targeted cleanup: `docker image prune` (just dangling images) or `docker container prune` (just stopped containers)
- Before removing volumes, check contents: `docker run --rm -v volume_name:/data alpine ls /data`
- Use `kubectl drain` before deleting nodes to gracefully reschedule workloads

### Cloud

| Pattern | Example | Risk |
|---------|---------|------|
| `terraform destroy` | `terraform destroy` | Tears down infrastructure |
| `aws s3 rm --recursive` | `aws s3 rm s3://bucket/ --recursive` | Deletes all objects in bucket |
| `aws ec2 terminate-instances` | `aws ec2 terminate-instances --instance-ids i-123` | Permanently destroys instance |
| `gcloud compute instances delete` | `gcloud compute instances delete my-vm` | Destroys VM and local disks |
| `az group delete` | `az group delete --name my-rg` | Deletes entire resource group |

**Safe alternatives:**
- Before `terraform destroy`, run `terraform plan -destroy` to preview what would be removed
- Before `aws s3 rm --recursive`, use `aws s3 ls` to verify bucket contents and confirm the correct bucket
- Enable versioning on S3 buckets so deletions are reversible
- Use `--dry-run` flags where available
- For cloud deletions, verify the account/project context first: `aws sts get-caller-identity`, `gcloud config get-value project`

## Production environment detection

Watch for signals that indicate you are working in a production or live environment. When any of these signals are present, apply extra caution to all operations, not just the patterns above.

**Environment variables:**
- `NODE_ENV=production`, `RAILS_ENV=production`, `FLASK_ENV=production`, `APP_ENV=production`
- `DATABASE_URL` or `DB_HOST` containing: `prod`, `production`, `live`, `primary`, `master` (as in primary DB)
- `AWS_PROFILE` or `AWS_ACCOUNT_ID` matching known production accounts
- `KUBECONFIG` or `kubectl config current-context` pointing to production clusters

**Hostnames and URLs:**
- Hostnames without `staging`, `dev`, `test`, `local`, `sandbox` qualifiers
- Database connection strings pointing to RDS, Cloud SQL, or other managed database endpoints
- API endpoints without `/staging/` or `/sandbox/` in the path

**File paths and configs:**
- `.env.production`, `config/production.yml`, `docker-compose.prod.yml`
- Deployment manifests referencing production namespaces

When you detect a production context, prefix your response with a clear notice: "I detect this is a production environment because [reason]. Proceeding with extra caution." Then apply stricter judgment to all commands, not just the ones in the pattern tables.

## When to warn vs. proceed

**Always warn (require explicit user confirmation):**
- Any command matching a dangerous pattern above that is not in the safe exceptions list
- Any destructive operation in a detected production environment
- Any operation you are uncertain about, where the downside is data loss

**Proceed without warning:**
- Commands matching the safe exceptions list (build artifact cleanup)
- Read-only operations (`SELECT`, `git log`, `kubectl get`, `docker ps`, `ls`)
- Operations in clearly local/development contexts with no production signals
- Non-destructive writes (creating files, `git commit`, `git push` without `--force`)

**When in doubt, warn.** A false alarm costs the user five seconds of reading. A missed warning can cost hours of recovery or permanent data loss. The cost asymmetry is extreme, so err heavily on the side of caution.

## How to warn

When you detect a dangerous pattern, format the warning clearly:

1. **State what you detected:** "This command uses `rm -rf` on a non-artifact directory."
2. **Explain the risk:** "This will permanently delete `/var/data` and all its contents."
3. **Suggest the alternative:** "Consider moving to trash first: `mv /var/data /tmp/trash-$(date +%s)/`"
4. **Ask for confirmation:** "Do you want to proceed with the original command, use the safe alternative, or cancel?"

Do not silently refuse. Do not proceed without asking. Always give the user the choice with full information.
